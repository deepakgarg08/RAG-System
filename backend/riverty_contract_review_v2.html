1. Problem Statement & What the System Solves

Riverty's legal team reviews hundreds of contracts annually. The current process is semi-manual — reviewers open each PDF individually, search for specific clauses, and record findings in spreadsheets. This creates three compounding problems:

Time Cost

Reading full PDFs clause by clause takes hours per contract. Business decisions are delayed waiting for legal sign-off.

Inconsistency

Different reviewers focus on different clauses. No consistent checklist means compliance risks slip through undetected.

No Cross-Document Search

"Which contracts have unlimited liability?" requires opening every file. There is no way to query the full portfolio simultaneously.

The brief: "Our legal colleagues regularly need to compare contracts and identify contracts that do not contain a certain text passage or contracts in which a company name needs to be updated. So far, this is semi-manual work."

The system has two completely independent pipelines that never interfere: the ETL pipeline runs once per document at ingest time; the RAG pipeline runs at query time. The LLM never sees a contract it does not need — it only sees the specific chunks retrieved for the current question, preventing hallucination by construction.

3. Three Analysis Modes

MODE 1 — Analyse Single Document

Upload any contract and ask questions about it. The document is extracted, analysed, and then immediately deleted — it is never added to the knowledge base. Useful for sensitive drafts not yet ready to store.

Also supports structured compliance check: returns a JSON result with compliant flag, violations list, and explanation.

POST /api/analyze · mode=single
POST /api/compliance

MODE 2 — Compare Against Knowledge Base

Upload a draft contract and compare it against all stored contracts. The system retrieves the most similar stored contracts semantically, then presents GPT-4o with both the uploaded document and retrieved contracts for a side-by-side comparison.

Identifies missing clauses, deviations, and company name mismatches.

POST /api/analyze · mode=compare

MODE 3 — Cross-Contract Query

Ask any question across all stored contracts simultaneously — no file upload needed. Uses the full hybrid retrieval pipeline (BM25 + semantic + reranking). The LangGraph agent classifies the query type and builds the appropriate context.

Answers include per-document verdicts for "which contracts" queries.

POST /api/query
Privacy guarantee — Mode 1 & 2: Temporary files are deleted in a finally block in the route handler — this is enforced at the server level, not just the UI label. Temporary documents never touch ChromaDB or any persistent storage.

Compliance Check (Mode 1)

A dedicated endpoint performs a structured compliance assessment against default legal guidelines. Returns machine-readable JSON, not a prose stream:

POST /api/compliance → ComplianceResult
{
  "compliant": false,
  "violations": [
    "No GDPR / data protection clause found",
    "Termination notice period not specified"
  ],
  "explanation": "The contract does not address data processing obligations..."
}

Default checks: GDPR / data protection clause · Termination rights and notice period · Governing law and jurisdiction · Liability cap and limitation clauses · Clear party identification

4. Data Ingestion Pipeline (ETL)

1

Extract

PDFExtractor (PyMuPDF): page-by-page text extraction, handles mixed English/German content. Falls back automatically to OCRExtractor (Tesseract 5 with German language model) if text extraction yields empty pages or for .jpg/.png files.

2

Transform

TextCleaner: normalises whitespace, pipe characters, and German umlauts.   LanguageDetector: tags each chunk as en or de.   ContentTypeDetector: classifies as qa / legal / narrative to select the appropriate chunking strategy.

Three chunking strategies: (1) QA: split at Q: markers — one chunk per question/answer pair. (2) Legal: split at §, Article N, Clause N — one chunk per clause (atomic unit of legal meaning). (3) Narrative: RecursiveCharacterTextSplitter at 1500 chars / 200-char overlap.

3

Embed & Load

EmbeddingService: generates 1024-dimensional vectors using BAAI/bge-m3 (multilingual, runs locally — no API call during ingest). Each chunk is stored in ChromaDB with 14 metadata fields: source file, page number, chunk index, total chunks, language, content type, MD5 checksum, file size, extraction method, is_scanned flag, embedding model, upload timestamp, text preview, chunking strategy.

4

Deduplication Guard

ingestion_registry.json tracks every ingested file by MD5 checksum. Uploading the same file twice returns status="skipped" instantly — no re-embedding. Attempting to ingest with a different embedding model returns 409 Conflict to prevent incompatible vector dimensions from mixing in the same collection.

5. Retrieval Layer — What's New

5.1 Classic Dense Retrieval (baseline)

The original retrieval pipeline used cosine similarity on BAAI/bge-m3 embeddings: the query is embedded to a 1024-dim vector, ChromaDB returns the top-K most similar chunks, and a threshold of ≥0.40 filters out low-relevance results. This works well for paraphrased queries but misses exact legal terms.

5.2 Hybrid BM25 + Dense + Reciprocal Rank Fusion NEW

Why hybrid matters: Dense vectors handle paraphrased queries ("ending the agreement" ↔ "termination"). BM25 handles exact legal terms that dense search can miss — entity names like "Riverty GmbH", article numbers like "§12", regulation citations like "Article 17 GDPR". RRF merges both ranked lists without requiring calibrated score scales.
# ============================================================ # DEMO MODE: BM25 index built in-memory over ChromaDB chunk texts. # PRODUCTION SWAP → Azure AI Search hybrid mode: # Azure AI Search runs BM25 + vector natively in a single API call. # No separate BM25 index needed — remove hybrid_retriever.py entirely. # ============================================================

The HybridRetriever class maintains an in-memory BM25Okapi index over all ChromaDB chunks. The index is built lazily on first query and automatically refreshed when new documents are ingested.

Reciprocal Rank Fusion (RRF) merges the two ranked lists using the formula:

RRF_score(chunk C) = Σ  1 / (K + rank_of_C_in_list_L)
                   L ∈ {dense, BM25}

K = 60 (standard constant — reduces impact of extreme top ranks)

Example: chunk "§12 Termination" ranked 1st in BM25 and 3rd in dense:
  RRF = 1/(60+1) + 1/(60+3) = 0.01639 + 0.01587 = 0.03226

Chunk appearing in only one list always scores lower than one appearing in both.
Fallback: if BM25 returns nothing → dense only; if dense returns nothing → BM25 only.

5.3 Cross-Encoder Reranking NEW

After RRF merging, the top candidates are reranked using cross-encoder/ms-marco-MiniLM-L-6-v2:

Bi-encoder (retrieval)Cross-encoder (reranking)
Encodes query and chunk independently, then compares embeddings. Fast — works at scale.Reads query and chunk text jointly in one forward pass. Much more accurate relevance signal.
May confuse "payment terms agreement" with "termination agreement" due to shared context.Correctly distinguishes fine-grained relevance — critical for short legal clauses.

The cross-encoder model (~80 MB) is lazy-loaded on first query and cached. If unavailable (e.g. no internet on first run), the system falls back silently to sorting by cosine similarity — no error, retrieval continues to work.

5.4 MMR Diversity Filter NEW

After reranking, Maximal Marginal Relevance (MMR) selects a diverse subset of top-K chunks. Without it, the top 8 chunks might all be slight variations of the same termination clause from the same document — wasting the LLM context window.

MMR_score(candidate C) = λ × relevance(C)  −  (1−λ) × max_similarity(C, already_selected)

λ = 0.6 → 60% relevance weight / 40% diversity penalty
Inter-chunk similarity: Jaccard overlap of token sets (no extra model needed)

Result: diverse set of chunks from different contracts and different clause types
        → GPT-4o receives broader evidence → better answer quality

6. LangGraph Agent Pipeline

The query pipeline is implemented as a 4-node LangGraph state machine. Each node receives the typed state dict and returns an updated version — independently testable and replaceable.

Node 1 · query_router

Classifies the question: find_clause, compare_contracts, summarise, general_question, find_missing. Determines context-building strategy in Node 3.

Node 2 · retriever

Runs HybridRetriever → CrossEncoder → MMR. Returns top-8 diverse, reranked chunks with full metadata (source file, page, chunk index, similarity score).

Node 3 · reasoner

For find_missing: groups chunks by document → per-document context → per-document verdict. Otherwise: flat context string → grounded-answer prompt → GPT-4o streaming.

Node 4 · formatter

Appends source citations with relevance dots: ● = high relevance (≥0.65), ◐ = relevant (0.40–0.65). Every claim is traceable to a specific document and page.

Why LangGraph over a simple chain: The query router means different query types get different context-building strategies without a monolithic prompt. Each node is independently unit-tested. Future additions (hallucination check node, feedback loop node) can be inserted without refactoring the existing graph.

10. Evaluation Framework — 7 Categories, 22 Metrics NEW

Design principle: Generic NLP metrics (BLEU, ROUGE) measure word overlap. A system can score well on BLEU and still fail to identify a missing GDPR clause. The 22 metrics here are designed specifically for RAG systems operating in the legal domain — each category catches a different class of failure.

Overview — All 22 Metrics at a Glance

1 · Retrieval (3 metrics)

  • Precision@K — fraction of retrieved chunks that are relevant
  • Recall@K — fraction of relevant chunks that were retrieved
  • Mean Reciprocal Rank (MRR) — average of 1/rank of first relevant result

2 · Generation (4 metrics)

  • Faithfulness — answer stays grounded in retrieved chunks (no hallucination)
  • Answer Relevance — answer actually addresses the question asked
  • Completeness — answer covers all expected points from ground truth
  • Hallucination Rate — fraction of claims unsupported by context

3 · Legal Domain (4 metrics)

  • Contract Hit (C-Hit) — Mode 2: does the answer reference the correct contract?
  • Keyword Hit (Kw-Hit) — expected legal keywords present in answer?
  • Absent Keyword Pass (AbsKw) — system correctly says clause is absent?
  • Comparison Hit (Cmp-Hit) — comparison answer covers both contracts?

4 · Context Quality (2 metrics)

  • Context Precision — are retrieved chunks actually useful to GPT-4o?
  • Context Recall — were any relevant chunks missed during retrieval?

5 · Attribution (2 metrics)

  • Citation Accuracy — cited source files actually exist in the knowledge base?
  • Citation Coverage — fraction of answer sentences that have at least one citation?

6 · Robustness (2 metrics)

  • Query Consistency — consistent answers across paraphrased versions of the same question?
  • Noise Sensitivity — answer stability when irrelevant noise chunks are injected?

7 · System Performance (2 metrics)

  • Latency — wall-clock time from request to final SSE token (milliseconds)
  • Cost per Query — estimated OpenAI token cost per query (USD)

Retrieval Metrics (Category 1) — Using the Eval Endpoint

IR metrics are computed via POST /api/eval/retrieve which runs HybridRetriever → CrossEncoder → MMR and returns raw chunks. This evaluates real production retrieval quality cheaply — no LLM token costs.

MetricFormula
Precision@K|{retrieved} ∩ {relevant}| / K
Recall@K|{retrieved} ∩ {relevant}| / |{relevant}|
MRRMean over queries of 1 / rank(first relevant chunk)

Faithfulness — Dual Backend (automatic fallback)

Primary — LLM Judge (GPT-4o-mini)

GPT-4o-mini receives: the question, retrieved context, and the system's answer. It scores faithfulness 0–10 based on whether every factual claim in the answer appears in the provided context. Score normalised to 0–1. Most accurate method.

Requires: OPENAI_API_KEY

Fallback — Local Embedding Cosine

Mean cosine similarity between context chunk embeddings and the answer embedding, computed locally using BAAI/bge-m3. Always available offline — no API call required. Lower absolute accuracy but always available.

Automatic fallback if API key absent

Legal Domain Metrics (Category 3) — Why They Matter

These are the most important metrics for Riverty's legal use case. Each maps directly to a real workflow scenario:

MetricModeReal scenario it validates
Contract Hit2When comparing a draft, does the system find and reference the correct comparison contracts?
Keyword Hit1, 2, 3When asked about GDPR, termination, or liability, does the answer contain the expected legal terms?
Absent Keyword Pass1When a clause is genuinely missing from a contract, does the system correctly report "not found" rather than hallucinating a clause?
Comparison Hit3For "which contracts differ in X", does the answer correctly identify all relevant contracts — not just the most similar one?

Robustness Testing

Query Consistency: The generate_paraphrases() function creates semantically-equivalent variants of each test question. The system must give consistent answers regardless of phrasing — critical because different legal reviewers may phrase the same question differently.

Noise Sensitivity: The inject_noise_chunks() function appends irrelevant distractor chunks to the retrieved context. A robust system should maintain its answer quality; a fragile one will be thrown off by irrelevant material. Low noise sensitivity score = robust.

11. How to Run & Evaluate

Backend Setup (5 minutes)

cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
# Set OPENAI_API_KEY in .env

uvicorn app.main:app --reload --port 8000
# → http://localhost:8000/docs

Frontend Setup

cd frontend
npm install && npm run dev
# → http://localhost:5173

Ingest Contracts

# Via UI: drag PDF to left sidebar
# Via API:
curl -X POST http://localhost:8000/api/ingest \
     -F "file=@contract.pdf"

# Batch ingest:
bash backend/run_pipeline.sh --ingest-only

Run Tests

cd backend
bash run_tests.sh
# → 122 tests, 0 failed

# Run evaluation harness:
python tests/eval/run_eval.py \
  --modes 1,2,3 --top-k 8 \
  --output-dir results/
Note on the evaluation harness: The backend server must be running before executing run_eval.py. Faithfulness scoring uses GPT-4o-mini (requires OPENAI_API_KEY). If the key is absent, the harness automatically switches to the local embedding fallback — all other metrics still compute normally.
