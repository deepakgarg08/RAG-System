1. What is this project?

Riverty Contract Review is an AI-powered legal contract analysis system. It allows Riverty's legal team to:

  • Upload contracts (PDF, scanned images) to a persistent knowledge base
  • Ask plain-English questions: "Does this contract have a GDPR clause?"
  • Compare a new contract against every existing contract in the knowledge base
  • Detect missing or risky clauses across all stored contracts
  • Get compliance check results in structured form (compliant / violations list / explanation)

The system runs in demo mode (on a laptop, no cloud required) and is architected to swap to Azure production services with minimal code change. Every demo component has a documented Azure equivalent.

2. Problem Statement

Riverty's legal team reviews hundreds of contracts annually. The manual review process has three pain points:

Pain PointImpact
Time — reading full PDFs clause by clauseHours per contract, delays business decisions
Inconsistency — different reviewers focus on different clausesCompliance risks slip through
No cross-contract search — can't ask "which contracts have unlimited liability?"Can't audit the full portfolio quickly

What the system solves:

  • Instant answers — ask a plain-English question, get an answer with source citations in seconds
  • Compliance check — automated structured check against GDPR, termination, liability, governing law
  • Portfolio analysis — hybrid search (semantic + keyword) across all stored contracts to find patterns and gaps

Why these specific choices matter for Riverty

  1. Azure-first architecture: Every demo component has a documented Azure production swap. Riverty is a Microsoft shop — this alignment was a design constraint from day one.
  2. Offline embeddings: BAAI/bge-m3 runs locally so contract text never leaves the machine during the demo. In production, Azure OpenAI keeps data within the Azure tenant.
  3. German + English: bge-m3 is multilingual; langdetect tags every chunk; Tesseract is installed with German language data. Riverty operates in Germany.
  4. Source citations: RAG (not fine-tuning) means every answer is traced to a specific document, page, and chunk — a legal requirement.
  5. Hybrid retrieval: BM25 handles exact legal terms ("§12", entity names) that dense vectors miss. RRF merge requires no calibration between score scales.

7. Three Analysis Modes

ModeNameData SourceBackend RouteWhen to use
MODE 1Analyze DocumentUploaded file only — never storedPOST /api/analyze (mode=single)"Review this new contract before signing"
MODE 2Compare with DBUploaded file + ChromaDBPOST /api/analyze (mode=compare)"How does this draft compare to our existing contracts?"

MODE 1 — Single Document Analysis

  • User uploads a PDF; text extracted to temp file, immediately deleted after extraction
  • File content is never written to ChromaDB
  • GPT-4o answers using only the uploaded document as context
  • Also supports compliance check (POST /api/compliance) → returns structured JSON: {"compliant": false, "violations": [...], "explanation": "..."}

MODE 2 — Compare with Database

  • User uploads a PDF (also never stored permanently)
  • System embeds the first 500 chars as a semantic query
  • Retrieves the most similar stored contracts from ChromaDB
  • GPT-4o sees both the uploaded document AND the retrieved contracts
  • Produces a comparison: "Your new contract differs from existing ones in these ways..."

MODE 3 — Cross-DB Query

  • No file upload needed; user asks a question about the stored contract portfolio
  • LangGraph's query_router classifies the query
  • HybridRetriever (BM25 + dense + RRF) fetches relevant chunks NEW
  • If find_missing: uses document_grouper to build per-document context → LLM gives per-document verdicts
  • If find_clause/compare_contracts etc.: flat retrieval → standard grounded answer

9. ETL Pipeline Deep Dive

Extractors (Strategy Pattern)

BaseExtractor (abstract)
    └── PDFExtractor   → PyMuPDF: page-by-page text, handles mixed EN/DE text
    └── OCRExtractor   → pytesseract: converts rasterized PDFs and images

The pipeline selects the extractor based on file extension: .pdf → PDFExtractor (if text is extractable) - .jpg, .jpeg, .png → OCRExtractor - Scanned PDFs → OCRExtractor (detected by empty text extraction)

Chunking Strategies

Three content types → three chunking strategies:

  • QA documents (FAQ contracts, questionnaires): Split at Q: markers. One chunk = one question + answer pair. Preserves semantic unity of each Q&A.
  • Legal documents (contracts, agreements): Split at §, Article N, Clause N, SECTION N headers. One chunk = one legal clause/article. Preserves the atomic unit of legal meaning.
  • Narrative documents (everything else): RecursiveCharacterTextSplitter(chunk_size=1500, chunk_overlap=200). Tries \n\n → \n → . before hard character split. 200-char overlap prevents sentences being cut at boundaries.

Chunk Metadata (14 fields)

Every stored chunk carries:

{
    "source_file": "contract_2024.pdf",
    "page_number": 3,
    "chunk_index": 2,
    "total_chunks": 8,
    "language": "en",            # or "de"
    "content_type": "legal",     # qa / legal / narrative
    "checksum": "a3f2...",       # MD5 of original file
    "file_size_kb": 142,
    "extraction_method": "pdf",  # or "ocr"
    "is_scanned": False,
    "embedding_model": "BAAI/bge-m3",
    "upload_timestamp": "2026-03-30T14:22:00Z",

11. NEW: Hybrid Retriever — BM25 + Dense + RRF NEW

Why hybrid matters for legal contracts: Dense vectors handle paraphrased queries ("ending the agreement" ↔ "termination"). BM25 handles exact legal terms that dense search can miss — "Riverty GmbH", "§12", "Article 17 GDPR". RRF merges both without requiring comparable score scales.
# ============================================================ # DEMO MODE: BM25 index built in-memory over ChromaDB chunk texts. # PRODUCTION SWAP → Azure AI Search hybrid mode (AWS: OpenSearch hybrid): # Azure AI Search runs BM25 + vector natively in a single API call. # No separate BM25 index needed — remove this file and update retriever.py. # ============================================================

Architecture

class HybridRetriever:
    # In-memory BM25Okapi index — rebuilt lazily when collection count changes
    # ChromaDB cosine similarity for dense search
    # RRF_K = 60 (standard constant)

    def retrieve(query, top_k=8):
        candidates = top_k * 2  # cast wide net before merging

        dense_results  = _dense_retrieve(query, candidates)
        bm25_results   = _bm25_retrieve(query, candidates)
        merged_results = _rrf_merge(dense_results, bm25_results, top_k)
        return merged_results

Reciprocal Rank Fusion (RRF)

RRF combines two ranked lists without needing calibrated scores between them:

RRF score for chunk C = Σ over each list L of: 1 / (K + rank_of_C_in_L)

Example (K=60):
  Chunk "§12 Termination" ranked 1st in BM25, 3rd in dense:
    RRF = 1/(60+1) + 1/(60+3) = 0.01639 + 0.01587 = 0.03226

  Chunk "cancellation clause" ranked 2nd in dense only:
    RRF = 1/(60+2) = 0.01613

  → "§12 Termination" wins because it appears in both lists.

BM25 Index Management

  • The BM25 index is built lazily on first query and cached in memory
  • Rebuilt automatically when ChromaDB collection count changes (new ingest detected)
  • Tokenization: simple whitespace split (lowercase) — sufficient for legal clause matching
  • Empty collection: BM25 returns empty, system falls back gracefully to dense-only

Fallback logic

  • If BM25 returns nothing → return dense results only (top_k)
  • If dense returns nothing → return BM25 results only (top_k)
  • If both return results → full RRF merge

12. NEW: Cross-Encoder Reranker & MMR Filter NEW

Cross-Encoder Reranking

After HybridRetriever returns candidate chunks, the cross-encoder reranker improves precision:

Model: cross-encoder/ms-marco-MiniLM-L-6-v2  (~80 MB, cached after first download)

# Bi-encoder (used in retrieval):
  query_embedding = embed(query)        # independent
  doc_embedding   = embed(chunk_text)   # independent
  score = cosine(query_embedding, doc_embedding)

# Cross-encoder (used in reranking):
  score = model.predict([(query, chunk_text)])  # joint scoring
  # Reads both texts together — much more accurate relevance

Why it matters: Short legal clauses often share surface form with many queries. A bi-encoder may score "The agreement may be terminated..." highly for both "termination notice period" and "payment terms" because both mention agreements. A cross-encoder reading them jointly correctly distinguishes them.

Fallback: If the cross-encoder model cannot be loaded (e.g. no internet on first download), the system falls back silently to sorting by similarity_score — the retrieval pipeline continues to work.

Maximal Marginal Relevance (MMR) Filter

After reranking, MMR selects a diverse subset of chunks to prevent the LLM seeing the same clause repeated 8 times:

MMR score for candidate C = λ × relevance(C) - (1-λ) × max_similarity(C, selected)

Parameters:
  λ = 0.6  →  60% relevance weight, 40% diversity penalty
  Inter-chunk similarity: Jaccard overlap of token sets (no extra model needed)

Algorithm: greedy iterative selection
  1. Start with empty selected set
  2. Pick chunk with highest MMR score from remaining candidates
  3. Add to selected, repeat until top_k chunks selected

Result: the LLM receives a diverse set of chunks covering different contracts and different aspects of the query, improving answer quality and reducing redundant context.

Query
  │
  ▼
HybridRetriever.retrieve(query, top_k=16)
  ├── BM25Okapi.get_scores(tokenized_query) → top-16 keyword matches
  ├── ChromaDB cosine search → top-16 dense matches
  └── RRF merge → top-16 combined candidates

13. LangGraph Agent — 4-Node Graph

State: {question, query_type, retrieved_chunks, answer}

query_router → retriever → reasoner → formatter
     │              │           │           │
 classifies    HybridRetriever  GPT-4o   add source
 query type    + reranker     streaming   citations
               + MMR

Node 1 — query_router: Classifies the question into one of: find_clause, compare_contracts, summarise, general_question, find_missing. This changes how the context is built in the reasoner.

Node 2 — retriever: Runs HybridRetriever → CrossEncoder → MMR (as described in Section 12). Returns top-8 diverse, reranked chunks with all metadata.

Node 3 — reasoner:

  • If find_missing: groups chunks by document, builds per-document context, uses a system prompt that asks GPT-4o to state per-document verdicts
  • Otherwise: builds flat context string, uses standard grounded-answer system prompt

Node 4 — formatter: Appends clean source citations with relevance dots:

  • ● = 0.65–1.0 (strongly relevant)
  • ◐ = 0.40–0.65 (relevant)
  • ○ = below threshold (filtered before reaching formatter)

Streaming

All three modes use Server-Sent Events (SSE):

POST /api/query or /api/analyze
→ StreamingResponse(media_type="text/event-stream")
→ Each token: "data: token\n\n"
→ End: "data: [DONE]\n\n"

React reads with fetch() + response.body.getReader() (not EventSource) — allows POST with body.

17. NEW: Evaluation Framework — 7 Categories, 22 Metrics NEW

Why domain-specific metrics matter: Generic NLP metrics (BLEU, ROUGE) measure word overlap. They don't tell you whether the system correctly identified a missing GDPR clause. Contract Hit, Keyword Hit, and Absent Keyword Pass measure exactly that — and can catch failures that BLEU would score as "passing".

Overview — All 22 Metrics

1. Retrieval (3 metrics)

  • Precision@K (P@K) — fraction of retrieved chunks that are relevant
  • Recall@K (R@K) — fraction of relevant chunks that were retrieved
  • Mean Reciprocal Rank (MRR) — average of 1/rank of first relevant result

2. Generation (4 metrics)

  • Faithfulness (Faith[src]) — how well answer stays grounded in retrieved chunks
  • Answer Relevance (Ans-Rel) — does the answer actually address the question?
  • Completeness Score (Complete) — does the answer cover all expected points?
  • Hallucination Rate (Halluc) — fraction of claims not supported by retrieved context

3. Task-Specific / Legal Domain (4 metrics)

  • Contract Hit (C-Hit) — Mode 2: does answer reference the correct contract? (binary)
  • Keyword Hit (Kw-Hit) — Modes 1,2,3: do expected keywords appear in the answer?
  • Absent Keyword Pass (AbsKw) — Mode 1: does the system correctly say a keyword is absent?
  • Comparison Hit (Cmp-Hit) — Mode 3: does the comparison answer both contracts correctly?

4. Context Quality (2 metrics)

  • Context Precision (Ctx-Prec) — are retrieved chunks actually useful to GPT-4o?
  • Context Recall (Ctx-Rec) — did we miss any relevant chunks that should have been retrieved?

5. Attribution (2 metrics)

  • Citation Accuracy (Cit-Acc) — are cited source files actually real files in the KB?
  • Citation Coverage (Cit-Cov) — what fraction of the answer's claims have citations?

6. Robustness (2 metrics)

  • Query Consistency (Consist) — does system give consistent answers to paraphrased queries?
  • Noise Sensitivity (NoiseSens) — how much does answer change when noise chunks are injected?

7. System Performance (2 metrics)

  • Latency (Lat) — wall-clock time from request to final SSE token (ms)
  • Cost per Query (Cost) — estimated OpenAI token cost per query (USD)

Metric Implementations — Pluggable Module Architecture

backend/tests/eval/

├── ground_truth.json    # Expected answers per test case
└── metrics/             # Pluggable metric modules

Category 1: Retrieval Metrics (IR metrics)

Computed via POST /api/eval/retrieve — returns raw chunks without LLM answer generation:

MetricFormulaRequires
Precision@K|relevant ∩ retrieved| / KGround-truth relevant chunk IDs per question
Recall@K|relevant ∩ retrieved| / |relevant|Ground-truth relevant chunk IDs per question
MRRmean over queries of 1/rank(first relevant chunk)Ground-truth relevant chunk IDs per question

Category 2: Generation Metrics

Faithfulness — two backends (automatic fallback)

  1. LLM judge (primary): GPT-4o-mini is given the question, retrieved context, and the system's answer. It scores faithfulness 0–10 with a structured prompt asking: "Does every factual claim in the answer appear in the provided context?" Score normalised to 0–1.
  2. Local embedding cosine (fallback): If OPENAI_API_KEY is absent, computes mean cosine similarity between context chunk embeddings and the answer embedding using BAAI/bge-m3. Always available offline.

Answer Relevance

GPT-4o-mini generates N back-questions from the system's answer, then measures cosine similarity between each back-question embedding and the original question embedding. High similarity → answer actually addresses the question asked.

Completeness Score

For each expected keyword/concept in the ground truth, checks whether it appears in the system's answer (case-insensitive, with synonym matching). Completeness = matched_concepts / total_expected_concepts.

Hallucination Rate

Inverse of faithfulness — fraction of claims in the answer that cannot be traced to any retrieved chunk. Uses the same LLM judge or embedding fallback as faithfulness.

These measure the quality of retrieved context, not the generated answer:

MetricHow computed
Context PrecisionFraction of retrieved chunks that GPT-4o-mini judges as "directly useful" for answering the question. High precision means the retriever wastes fewer LLM context window tokens on irrelevant material.
Context RecallFraction of ground-truth relevant information that appears in the retrieved context. Low recall means the system is missing crucial clauses even though it might answer partially correctly.

Category 5: Attribution Metrics

MetricHow computed
Citation Accuracy (Cit-Acc)Parses source citations from the answer (filename · page · chunk format). Checks each cited source_file exists in the ingested KB. Score = valid_citations / total_citations.
Citation Coverage (Cit-Cov)Ratio of answer sentences that have at least one citation. High coverage = every claim is traceable. Critical for legal audit requirements.

Category 6: Robustness Metrics

Query Consistency (Consist)

The generate_paraphrases() function creates N semantically-equivalent versions of each test question. The same question is then sent as each paraphrase. Consistency = similarity between all N answers (measured by pairwise cosine similarity of answer embeddings). A low score means the system gives inconsistent answers to the same question worded differently.

Noise Sensitivity (NoiseSens)

The inject_noise_chunks() function appends irrelevant distractor chunks to the retrieved context before passing to the LLM. The score_noise_sensitivity() function measures how much the answer changes relative to the clean-context answer. Low sensitivity = robust to irrelevant context.

Category 7: System Performance

MetricHow measured
Latency (Lat)Wall-clock time from POST request sent to final SSE [DONE] token received. Measured with time.perf_counter(). Includes all retrieval, reranking, LLM generation, and SSE streaming time.
Cost per Query (Cost)estimate_query_cost() in cost_metrics.py estimates input/output token counts and multiplies by current GPT-4o pricing. aggregate_session_cost() sums across all eval queries.

19. Interview Q&A — Every Question They Could Ask

Architecture & Design

Q: Why RAG instead of fine-tuning an LLM on contract data?

A: Three reasons. First, contracts change frequently — a new contract is queryable seconds after upload; fine-tuning requires a full retraining cycle. Second, legal requirement: every answer must cite the source clause — RAG gives this automatically through chunk metadata; fine-tuning cannot. Third, GDPR: storing contract content in model weights is hard to audit and impossible to delete (right to erasure requires deletion from the vector store entry, not retraining the model).

Q: Why LangGraph instead of a simple LLM call?

A: LangGraph gives us a typed state machine. Each of the four nodes (router, retriever, reasoner, formatter) is independently testable, replaceable, and observable. The query router node means we handle different query types differently — "find missing clauses" uses document-grouped context; "find a specific clause" uses flat retrieval. LangGraph also enables conditional branching and future additions (e.g., a reranker node, a hallucination check node) without refactoring the whole pipeline.

Q: Why Server-Sent Events instead of WebSocket?

A: SSE is strictly server-to-client, which perfectly matches our use case — the client sends one question and only needs to receive a streamed answer. SSE works through all HTTP proxies and corporate firewalls as plain HTTP with no protocol upgrade. WebSocket adds bidirectional complexity (ping/pong, reconnect logic) with zero benefit here. FastAPI's StreamingResponse supports SSE natively.

Q: Why ChromaDB for demo and Azure AI Search for production?

A: ChromaDB requires zero configuration — no cloud account, no API keys beyond OpenAI, no separate service to start. It runs in-process with Python and persists to local disk. A reviewer can clone the repo, set one API key, and have the full system running in 5 minutes. Azure AI Search adds what ChromaDB lacks in production: hybrid vector + keyword search (better recall on short legal terms), enterprise RBAC, horizontal scaling across pods, and SOC 2 / ISO 27001 compliance. The swap is one line: ChromaLoader() → AzureSearchLoader().

Q: Why add BM25? Isn't dense vector search enough? NEW

A: Dense vectors are excellent at paraphrase matching ("ending the agreement" ↔ "termination") but weak on exact term matching. Legal contracts contain critical exact terms: specific article numbers (§12), entity names ("Riverty GmbH"), regulation citations ("Article 17 GDPR"). A dense model may embed "§12" similarly to many other paragraph references, losing the specificity. BM25 gives exact keyword matching. RRF merges both without requiring calibrated score scales — a chunk that appears in both ranked lists scores higher than one that appears in only one. In production, Azure AI Search provides this hybrid natively, so no additional code is needed.

Q: Why cross-encoder reranking? NEW

A: Bi-encoders (used in retrieval) encode the query and each chunk independently, then compare embeddings. This is fast but misses fine-grained relevance signals. A cross-encoder reads the query and chunk text jointly — it can catch that "the agreement terminates" is highly relevant to "termination notice period" but not to "payment terms", even though both contain "agreement". The model used (ms-marco-MiniLM-L-6-v2) is ~80MB and runs on CPU. The cross-encoder scores override the retrieval scores, improving precision at the top of the final ranked list presented to GPT-4o.

Q: Why MMR diversity filtering? NEW

A: Without diversity filtering, the top-8 retrieved chunks might all be slight variations of the same termination clause from the same document — wasting the LLM's context window and producing a narrow, repetitive answer. Maximal Marginal Relevance iteratively selects the next chunk that maximises relevance while penalising similarity to already-selected chunks. The λ=0.6 parameter keeps 60% relevance weight — we prioritise quality but actively remove redundancy. This means GPT-4o sees diverse evidence from different contracts and different clause types.

Q: Why BAAI/bge-m3 instead of OpenAI text-embedding-3-large for the demo?

A: Three reasons. First, cost: bge-m3 is free — no API calls during ingestion, which is the most expensive phase. Second, privacy: contract text never leaves the machine during embedding — only GPT-4o queries hit an external API. Third, multilingual: bge-m3 handles German and English in one model without switching. In production we switch to Azure text-embedding-3-large because Azure OpenAI keeps data within the Microsoft tenant, which is required for Riverty's real contracts.

Q: How does the system prevent hallucination?

A: The reasoner node's system prompt explicitly instructs GPT-4o: "Answer only from the provided contract excerpts. If the information is not in the excerpts, say so." The formatter node appends source citations with relevance scores — users can verify every claim against the original document. The similarity threshold (0.40) filters out low-relevance chunks before they reach GPT-4o, so the model sees only genuinely relevant context. We do not prevent all hallucination (no system does) but we make it auditable.

Evaluation Framework NEW

Q: Why 22 metrics? Isn't accuracy enough?

A: "Accuracy" is undefined for open-ended RAG systems. The 22 metrics cover different failure modes that single-number accuracy cannot capture: a system can score well on BLEU but still fail to identify a missing GDPR clause (legal domain metric failure). It can retrieve relevant chunks but present them in a confusing order (context quality metric failure). It can give correct answers but without citations (attribution metric failure). Each category catches a different kind of failure. In practice, for a production deployment we'd focus on 3–5 key metrics per mode, but having all 22 defined lets us diagnose root causes precisely.

Q: What is faithfulness and how do you measure it without RAGAS?

A: Faithfulness measures whether the generated answer is grounded in the retrieved context — i.e., no claims beyond what the context provides. We use a two-tier approach: (1) GPT-4o-mini as an LLM judge — it reads the question, context, and answer and scores on a 0–10 scale how well the answer stays within the provided excerpts. This is the most accurate method. (2) Local embedding cosine similarity — if OpenAI API is unavailable, we compute mean cosine similarity between context chunk embeddings and the answer embedding using bge-m3. This is an offline proxy. RAGAS is the production-grade path for both faithfulness and answer relevancy — integrating it is a one-sprint task.

Q: How does query consistency testing work?

A: The generate_paraphrases() function creates semantically-equivalent question variants. For "Does contract A have a GDPR clause?" it might generate "Is there a data protection provision in contract A?" and "Does contract A address GDPR requirements?". We send all variants to the system and compare answers using pairwise cosine similarity of answer embeddings. A high consistency score (≥0.85) means the system gives the same answer regardless of phrasing. This is particularly important for legal use cases where different reviewers may phrase the same question differently.

Production & Scaling

Q: How would this scale to 10,000 contracts?

A: The architecture is designed for this. Swap ChromaDB for Azure AI Search — it handles millions of vectors with sub-second query time. Swap the local embedding model for Azure OpenAI embeddings service — batching is already implemented. Add an Azure Service Bus queue between the upload API and the ETL pipeline so ingestion is asynchronous and doesn't block the HTTP response. The LangGraph agent is stateless — run multiple instances behind Azure API Management.

Q: How do you handle GDPR data subject requests (right to erasure)?

A: Each chunk's metadata includes the source filename and MD5 checksum. To delete a contract: remove its chunks from ChromaDB by querying for source_file == "contract.pdf", delete the raw file from Azure Blob Storage, and remove the registry entry. The LLM model weights are never updated — the contract text only ever lived in the vector store and blob storage. This is the key GDPR advantage of RAG over fine-tuning.

Q: What monitoring would you add for production?

A: Azure Monitor + Application Insights. Track: query latency (p50/p95/p99), retrieval hit rate (what fraction of queries get at least one chunk above threshold), LLM token usage (cost control), ingestion failures (by document type), compliance check throughput. Add distributed tracing across the LangGraph nodes so you can see which node is the bottleneck for a specific query. Run the evaluation harness weekly to catch model regressions after OpenAI model updates.

Q: What would you change if you had more time?

A: Four things in priority order. 1. Add GET /api/files listing endpoint so the sidebar persists across page refreshes. 2. Re-index ChromaDB with 1500-char chunks consistently (currently mixed 1000/1500 from different development phases). 3. Add streaming for the compliance check endpoint — currently it waits for the full response which can take 3–4 seconds for long contracts. 4. Add Azure Service Bus queue for async ingestion — currently the HTTP request blocks until ingestion completes, which takes 10–30 seconds for large PDFs.
