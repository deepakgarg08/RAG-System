Why this system exists

⏱

Manual Review

Hours per contract. Each reviewer opens the PDF, manually searches for clauses, and makes notes in a spreadsheet. No consistency.

🔍

No Cross-Document Search

"Which contracts have unlimited liability?" requires opening every file. There is no way to query across the corpus simultaneously.

⚠️

Inconsistent Reviews

Different reviewers miss different clauses. No audit trail. No confidence that every document was checked against the same standard.

Retrieval

Question embedded with same model as documents. Cosine similarity ≥ 0.40 threshold. Top-8 chunks from all contracts returned.

Grounding

System prompt: "Answer ONLY using the provided excerpts. If the information is not in these documents, say 'not found'." The LLM cannot invent clauses.

Source Attribution

Every answer ends with citations: filename · page number · chunk index · relevance dots (●◐○). Legal team can verify each claim directly.

Intelligent chunking strategy

Q&A Documents

Splits at Q: markers. One chunk = one question + answer pair.

Detected by: lines starting with "Q:"

Legal Contracts

Splits at §, Article N, Clause N, Artikel, Abschnitt. One chunk = one legal clause.

Detected by: §, Artikel, GDPR, Clause keywords

Narrative

RecursiveCharacterTextSplitter — 1500 chars, 200-char overlap so no sentence is cut at a boundary.

Default for all other documents

14 Metadata Fields Per Chunk

source_file page_number chunk_index language (en/de) content_type checksum (MD5) extraction_method is_scanned embedding_model upload_timestamp text_preview chunking_strategy
1 minute. "Three chunking strategies, auto-detected by content type. Legal contracts split at clause headers — one chunk per clause, which is the atomic unit of legal meaning. If a chunk contains half a clause and half the next one, the retrieval quality degrades significantly." "The ContentTypeDetector classifies each document automatically by looking for legal signal words — §, Artikel, GDPR, Clause — and QA patterns. For German contracts it recognises Abschnitt, Artikel, and Vertrag headers." "Every chunk carries 14 metadata fields. The checksum enables deduplication — upload the same file twice and it returns status=skipped. The language field means we never mix German and English chunks in a single answer without knowing which is which."

Evaluation framework — 7 categories, 22 metrics

1. Retrieval
  • Precision@K (P@K)
  • Recall@K (R@K)
  • MRR
2. Generation
  • Faithfulness
  • Answer Relevance
  • Completeness
  • Hallucination Rate
3. Legal Domain
  • Contract Hit (Mode 2)
  • Keyword Hit (1,2,3)
  • Absent Kw Pass (Mode 1)
  • Comparison Hit (Mode 3)
4. Context Quality
  • Context Precision
  • Context Recall
5. Attribution
  • Citation Accuracy
  • Citation Coverage
6. Robustness
  • Query Consistency
  • Noise Sensitivity
7. System Performance
  • Latency (ms)
  • Cost per Query

Why this matters: Domain-specific metrics (Contract Hit, Keyword Hit, Absent Keyword Pass) catch correctness failures that generic NLP metrics miss entirely. A system can score well on BLEU but still fail to identify a missing GDPR clause.

1.5 minutes. "We implemented a full evaluation framework — not just BLEU or accuracy, but 22 metrics across 7 categories designed specifically for RAG systems in the legal domain." "The key insight is Category 3 — Legal Domain metrics. Generic NLP metrics like BLEU or ROUGE measure word overlap. They don't tell you whether the system correctly identified a missing GDPR clause. Contract Hit, Keyword Hit, and Absent Keyword Pass measure exactly that." "Faithfulness measures whether the generated answer stays grounded in retrieved chunks — this is the hallucination metric. In production, we'd run this after every deployment to catch model regressions." "Context Precision tells us whether the chunks we retrieved were actually useful — if we're retrieving irrelevant chunks and GPT-4o is ignoring them, we're wasting tokens and adding latency." If asked about RAGAS: "RAGAS is the production-grade path for faithfulness and answer relevancy. We implemented the full metric specification; integrating with RAGAS for automated scoring is a one-sprint task."
